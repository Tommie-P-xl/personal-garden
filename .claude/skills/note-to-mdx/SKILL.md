# 手稿笔记转 MDX

将粗糙的学习手稿转换为结构化、组件丰富的 MDX 笔记文件，可直接放入 personal-garden 项目中使用。

## 核心原则

**使用者提供内容要点和框架，你负责结构优化和组件编排。**

不要求使用者了解 MDX 组件细节，你来判断哪些内容适合用什么组件展示。

## 工作流程

```
输入手稿 → 分析内容 → 规划章节 → 【询问用户】→ 确认结构 → 生成 MDX
                    ↑                              │
                    └──── 用户要求修改 ←────────────┘
```

### 第一步：读取与分析

读取用户提供的笔记内容，识别：
- **核心主题**：这篇笔记讲什么？
- **知识要点**：有哪些关键概念、步骤、公式、代码？
- **逻辑关系**：要点之间是什么关系？（并列、递进、对比、包含）
- **内容类型**：每个要点属于哪种类型？（概念解释、代码示例、公式推导、对比说明、操作步骤）

<Callout type="warning">
**用户要求记录：** 在这一步，必须记录用户提出的所有额外要求（如"代码注释要详细"、"多用图表"等）。这些要求必须贯穿后续所有步骤，不可遗忘。
</Callout>

### 第二步：章节规划

根据内容逻辑组织章节结构：

| 内容特征 | 推荐结构 |
|---------|---------|
| 概念+细节 | 概述 → 分章节详解 → 总结 |
| 步骤/流程 | 简介 → 步骤1/2/3 → 注意事项 |
| 对比类 | 概述 → 对比维度 → 各方案详述 |
| 知识树 | 总览（BraceMap）→ 分支详解 |

**章节顺序原则：**
1. 先总后分：先给全局视角，再深入细节
2. 先简后难：基础概念放前面，进阶内容放后面
3. 先因后果：先讲原理，再讲应用

### 第三步：询问用户文档组织方式（必须！）

**章节规划完成后，必须询问用户选择文档组织方式。** 这是强制步骤，不可跳过。

使用 AskUserQuestion 工具向用户提问，提供以下选项：

**选项 A：单文档模式**
- 所有内容写在一个 `.mdx` 文件中
- 适合内容较少、逻辑紧凑的主题
- 示例：`0001-Git基础.mdx`

**选项 B：多文档模式**
- 内容拆分为多个 `.mdx` 文件
- 可以利用子文件夹进行分层组织（最多 3 层嵌套）
- 适合内容丰富、主题可独立拆分的笔记系列

### 第四步：多文档结构规划（仅选项 B）

如果用户选择多文档模式，需要输出详细的目录结构供用户确认：

**结构规划原则：**
- 直接相关的章节放在同一目录下作为并行的 MDX 文件
- 内容较深、需要单独展开的章节可以新建子文件夹
- 子文件夹内可以有多个并行的 MDX 文件
- 子文件夹嵌套最多 3 层（含根目录）
- 每个新目录必须有 `_meta.json`

**输出格式示例：**

```
src/content/notes/0005-git/
├── _meta.json                          ← 目录元数据
├── 0001-核心概念.mdx                   ← 基础概念（独立文件）
├── 0002-环境配置与仓库管理.mdx          ← 配置+仓库操作
├── 0003-分支管理.mdx                   ← 分支相关操作
├── 0004-暂存与提交.mdx                 ← 工作区→暂存区→版本库
├── 0005-远程操作.mdx                   ← push/pull/fetch
├── 0006-撤销与回退.mdx                 ← reset/revert/reflog
├── 0007-标签管理.mdx                   ← Tag 操作
└── 0008-GitHub-Actions/               ← 子文件夹（内容较深，独立展开）
    ├── _meta.json
    ├── 0001-基础概念与Workflow结构.mdx
    ├── 0002-触发事件与条件执行.mdx
    ├── 0003-环境变量与Secrets.mdx
    └── 0004-矩阵构建与高级用法.mdx
```

**呈现给用户时，说明：**
1. 每个文件包含哪些章节（简要概述）
2. 为什么某些内容放在子文件夹中（内容深度/独立性）
3. 文件之间的逻辑关系

**等待用户确认后才能进入下一步。** 如果用户要求修改，调整结构后再次确认。

### 第五步：选择组件

根据内容类型选择最合适的组件（详见下方速查表）。

### 第六步：生成 MDX

逐个文件输出完整的 `.mdx` 内容，每个文件包含：
- 正确的 frontmatter
- 合理的章节结构
- 恰当的组件使用
- `[[双向链接]]` 关联相关笔记
- `<Progress>` 展示学习进度（如适用）

**生成过程中必须持续遵守用户提出的所有要求。** 例如：
- 用户要求"代码注释详细" → 每个代码块都必须有详细注释
- 用户要求"多用图表" → 积极使用表格、BraceMap 等可视化组件
- 用户要求"保持简洁" → 精简文字，避免冗余说明

## 用户要求贯穿机制

<Callout type="error" title="强制规则">
用户提出的要求必须贯穿整个生成过程，不可遗忘或忽略。
</Callout>

### 要求记录

在分析阶段，提取并记录用户的所有要求：
- 内容要求（如"代码注释要详细"、"多用示例"）
- 风格要求（如"保持简洁"、"通俗易懂"）
- 格式要求（如"多用表格对比"、"加上流程图"）
- 其他特殊要求

### 要求执行

在生成每个文件时，对照检查用户要求是否已落实：

| 用户要求 | 执行方式 |
|---------|---------|
| "代码注释详细" | 每个代码块都有行内注释，解释每行/每段的作用 |
| "多用图表" | 积极使用表格、BraceMap、对比表等可视化元素 |
| "保持简洁" | 精简文字描述，用要点列表代替长段落 |
| "加上示例" | 每个概念都配有实际示例 |
| "通俗易懂" | 使用类比、比喻，避免过多术语 |
| "对比说明" | 用表格或 Tabs 组件进行多方案对比 |

## 组件选择速查表

| 内容类型 | 推荐组件 | 使用场景 |
|---------|---------|---------|
| 重要提示/警告 | `<Callout>` | type: tip/info/warning/error |
| 定义/详细说明（可折叠） | `<Collapse>` | 不影响主流程的补充内容 |
| 多语言代码对比 | `<Tabs>` + `<TabItem>` | 同一功能的不同语言实现 |
| 数学公式 | `<Latex>` + `$$..$$` | 公式推导、数学模型 |
| 层次/树形结构 | `<BraceMap>` | 知识体系、目录结构、分类 |
| 对比数据/属性表 | Markdown 表格 | 简单对比，自动转换样式 |
| 复杂表格（需对齐/标题） | `<Table>` | 需要 align 或 caption 时 |
| 学习进度展示 | `<Progress>` | 标记各主题完成度 |
| 图片展示 | `![](path)` 或 `<Image>` | 需要调整大小/对齐时用组件 |
| 视频/文档 | `<Media>` | 视频播放或文档下载 |
| 多图展示 | `<Gallery>` | 图片网格布局 |
| 交叉引用 | `[[笔记标题]]` | 关联其他笔记 |

## 组件使用模板

### Callout 提示框
```mdx
<Callout type="tip">
这是实用技巧。
</Callout>

<Callout type="warning">
注意事项内容。
</Callout>

<Callout type="info">
补充说明内容。
</Callout>
```

### Collapse 折叠面板
```mdx
<Collapse title="展开查看详细内容">
详细说明或代码示例。
</Collapse>
```

### Tabs 标签页
```mdx
<Tabs labels={['标签1', '标签2', '标签3']}>
<TabItem index={0} active>
第一个标签的内容。
</TabItem>
<TabItem index={1}>
第二个标签的内容。
</TabItem>
<TabItem index={2}>
第三个标签的内容。
</TabItem>
</Tabs>
```

### Latex 公式
```mdx
<!-- 块级公式（默认居中） -->
<Latex>
$$
E = mc^2
$$
</Latex>

<!-- 左对齐 -->
<Latex align="left">
$$
\sum_{i=1}^{n} i = \frac{n(n+1)}{2}
$$
</Latex>

<!-- 行内公式直接写 $...$ -->
行内公式：$a^2 + b^2 = c^2$
```

### BraceMap 结构图
```mdx
<BraceMap
  data={{
    name: "根节点",
    children: [
      { name: "子节点1" },
      { name: "子节点2", children: [{ name: "孙节点" }] },
    ],
  }}
/>
```

### Progress 进度条
```mdx
<Progress value={80} label="主题名称" />
```

### 双向链接
```mdx
- [[相关笔记标题]] — 关联说明
```

<Callout type="error" title="重要：链接格式">
**双向链接必须使用笔记的 `title`（不含序号前缀），不是文件名！**

错误示例：`[[0001-核心概念]]`、`[[0003-暂存与提交]]`
正确示例：`[[核心概念]]`、`[[暂存与提交]]`

原因是系统会根据 `title` 来匹配笔记，使用带序号的文件名会导致链接失效。
</Callout>

## Frontmatter 规范

```yaml
---
title: 笔记标题
tags: [标签1, 标签2, 标签3]
summary: 一句话概括笔记内容（可选但推荐）
---
```

**标签命名建议：** 使用小写中文，2-4个标签为宜。

## 文件命名规范

```
src/content/notes/{分类目录}/{四位数序号}-{中文标题}.mdx
```

示例：`src/content/notes/0001-linux/0001-memory-management/0003-页面置换算法.mdx`

- 四位数序号控制同级内的排列顺序
- 渲染时自动去掉序号前缀
- 新建目录时必须同时创建 `_meta.json`

## 常见错误

| 错误 | 正确做法 |
|------|---------|
| 所有内容平铺，无组件 | 根据内容类型选择合适组件 |
| Callout 内容过长 | Callout 只放关键提示，详细内容用 Collapse |
| Tabs 用于非对比场景 | Tabs 用于同主题的多视角对比 |
| 忘记写 frontmatter | 每个 MDX 文件必须有 frontmatter |
| 新建目录不写 _meta.json | 新目录必须同时创建 _meta.json |
| 公式不用 Latex 组件包裹 | 块级公式用 `<Latex>` 包裹以控制对齐 |
| 序号格式错误 | 使用四位数：0000、0001、0013 |
| 标签过多或过少 | 2-4 个标签，覆盖主题和关键技术点 |
| **忽略用户要求** | **用户要求必须贯穿全程，每个文件都要检查** |
| **跳过询问步骤** | **章节规划后必须询问用户单文档/多文档** |
| **双向链接使用文件名** | **双向链接必须使用笔记的 `title`（不含序号前缀），不是文件名！** |

## 完整流程示例

**用户输入：**
```
我有一份 Git 学习笔记，内容如下：
[笔记内容...]
要求：代码注释要详细
```

**第一步：分析** → 识别核心主题、知识点、逻辑关系
→ 记录用户要求：代码注释详细

**第二步：章节规划** → 规划 12 个章节

**第三步：询问用户**（必须！）

使用 AskUserQuestion：
- 选项 A：单文档（所有内容在一个文件）
- 选项 B：多文档（拆分为多个文件，可含子文件夹）

**第四步（如选 B）：输出结构** → 等待用户确认

```
src/content/notes/0005-git/
├── _meta.json
├── 0001-核心概念.mdx
├── 0002-环境配置.mdx
├── 0003-分支管理.mdx
└── 0008-GitHub-Actions/
    ├── _meta.json
    ├── 0001-基础概念.mdx
    └── 0002-高级用法.mdx
```

**第五步：生成 MDX** → 每个文件都确保代码注释详细

## 使用方式

在 Claude Code 中直接描述你的需求：

```
/note-to-mdx

我有一份关于 TCP/IP 协议的学习笔记，内容如下：
[粘贴你的笔记内容]

请帮我转换为 MDX 格式，放到 src/content/notes/0005-network/ 目录下。
要求：代码注释要详细，多用表格对比。
```
